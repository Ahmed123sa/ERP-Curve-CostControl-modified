# قوانين — مرجع شخصي

## 1. هيكل المشروع
- **Backend**: Laravel 11 API (`ERP/laravel-app/`)
- **Frontend**: Next.js 14.2.0 (`ERP/frontend/`)
- **القاعدة**: MySQL (`erp_cost_control`)

## 2. أسماء الـ Variables في الداشبورد (KPI)
| الاسم العربي | API field | المصدر |
|---|---|---|
| إجمالي المشتريات | `total_purchases` | `stock_ledger` — `voucher_type='purchase' AND movement_type='in'` للمخازن main/sub بس |
| قيمة المنصرف | `total_dispatched` | `stock_ledger` — `voucher_type='dispatch' AND movement_type='out'` للمخازن main/sub بس (إجمالي اللي طلع من المخازن للفروع) |
| إجمالي الفروق | `total_diffs` | `monthly_closings` — مجموع `diff_value` للمخازن main/sub بس |
| قيمة المخزون (مخازن) | `total_stock_value_warehouses` | `monthly_closings` — مجموع `closing_value` للمخازن main/sub |
| قيمة المخزون (فروع) | `total_stock_value_branches` | `monthly_closings` — مجموع `closing_value` للمخازن type='branch' |

## 3. قوانين ثابتة (متفق عليها)
- **الفروق (diffs)**: للمخازن main/sub فقط. الـ type='branch' مش بيعمل diff (بيشتغل بـ "حرق خامات")
- **Food Cost %**: مش موجود في الداشبورد — من غير إيرادات مبيعات ملهاش معنى
- **قيمة المخزون (مخازن)**: `closing_qty × avg_cost` للمخازن main/sub — ومستحيل تكون سالب. لو ظهرت سالب، دا بسبب data issue
- **قيمة المخزون (فروع)**: `closing_qty_actual × avg_cost` للمخازن type='branch' — إجمالي قيم آخر المدة **الفعلي** مش النظري للفروع. وهتكون دايما بصفر طول مافيش آخر مدة فعلي (physical count) مدخل للفرع
- **closing_qty_theoretical**: دايمًا `MAX(0, opening + in - out)` — مخزون سالب ملهاش معنى فيزيائي
- **diff_value**: `(closing_qty_theoretical - closing_qty_actual) × avg_cost` — موجب = اوفر، سالب = عجز
- **closing_value**: `closing_qty_theoretical × avg_cost` — دايمًا >= 0

## 4. الفرق بين المخازن والفروع
- **main/sub**: بيجيلها مشتريات، صادر، فروق (عجز/اوفر) — المخزن الرئيسي والفرعي
- **branch**: بيجيلها وارد من المخازن (dispatch orders) — ملهاش diff، ملهاش مشتريات مباشرة

## 5. الـ Dispatch Order (إذن صرف فرع)
- بيحصل بين warehouse (main/sub) و branch
- في `stock_ledger`:
  - **OUT** من warehouse: `voucher_type='dispatch', movement_type='out'`
  - **IN** للـ branch: `voucher_type='dispatch', movement_type='in'`
- مجموع OUT = مجموع IN = قيمة المنصرف (dispatched total)

## 6. معادلات الحسابات الأساسية
```
closing_qty_theoretical = MAX(0, opening_qty + in_qty - out_qty)
closing_value           = closing_qty_theoretical × avg_cost
diff_qty                = closing_qty_theoretical - closing_qty_actual
diff_value              = diff_qty × avg_cost
avg_cost                = calcAvg > 0 ? calcAvg : defaultCost > 0 ? defaultCost : lastWeightedAvg
```

## 10. الأخطاء الشائعة
- `closing_qty_theoretical` سالب — دايمًا استخدم `MAX(0, ...)`
- `total_dispatched` يشمل production/withdrawal/external_sale — دايمًا فلتر بـ `voucher_type='dispatch'`
- `diff_value` في `monthly_closings` يمكن يكون stale لو `closing_qty_actual === null` — متفترضش إنه محدث
- مخازن type='branch' ملهاش diff — متفترضش إن الـ diff_value بتاعها صحيح

## 11. الـ Artisan Commands
| الأمر | الوظيفة |
|---|---|
| `php artisan closing:fix` | تصليح negative values في monthly_closings |

## 12. Flow: تصليح البيانات القديمة
1. `closing:fix` command — يلف على كل records ويعمل `MAX(0, closing_qty_theoretical)` ويعيد حساب `closing_value` و `diff_value`
2. لو `closing_qty_actual === null` — ميتغيرش diff (يخلي القديم)
3. ميمسحش records — الـ physical_count محفوظ
